# Research: Linux file change monitoring for a containerized PaaS rootfs persistence daemon

## Summary
A watcher-only persistence daemon cannot be 100% correct on Linux/Node: inotify and fanotify both have overflow/loss modes, race windows, stale path/name semantics, and explicit non-coverage cases. For a rootfs persistence daemon, use watchers as a low-latency hint stream, but pair them with durable snapshot/reconciliation, overflow-triggered full rescan, startup/shutdown scans, and idempotent copy/apply logic.

## Findings
1. **inotify is the baseline unprivileged Linux API, but it is not a recursive filesystem journal.** It monitors files or directories; directory watches report events for the directory and immediate children only, not the whole subtree. A tree watcher must add a watch for every directory and add watches for new subdirectories as they appear. [inotify(7)](https://man7.org/linux/man-pages/man7/inotify.7.html)
   - Practical implication: a rootfs daemon needs a maintained watch table plus recursive crawls. Initial scan and watch installation must be treated as a race-prone bootstrap, not a complete snapshot.

2. **Adding inotify watches requires read permission and consumes per-user kernel quota.** `inotify_add_watch()` requires read permission on the target; failures include `EACCES`, `ENOENT`, `ENOMEM`, and `ENOSPC` when the user watch limit or kernel resources are exhausted. The relevant tunables are `/proc/sys/fs/inotify/max_queued_events`, `max_user_instances`, and `max_user_watches`. [inotify_add_watch(2)](https://man7.org/linux/man-pages/man2/inotify_add_watch.2.html), [inotify(7)](https://man7.org/linux/man-pages/man7/inotify.7.html)
   - Practical implication: large PaaS root filesystems can hit watch limits. The daemon should preflight/tune limits, fail loudly, expose metrics, and fall back to crawl-only reconciliation if it cannot cover the tree.

3. **inotify overflow means events are lost.** When `max_queued_events` is exceeded, excess events are dropped and the kernel emits `IN_Q_OVERFLOW` with `wd == -1`; robust applications are expected to rebuild part or all of their cache. inotify also coalesces identical unread events, so it cannot reliably count file operations. [inotify(7)](https://man7.org/linux/man-pages/man7/inotify.7.html)
   - Practical implication: after any detected overflow, discard watcher-derived state and perform a full or bounded subtree rescan. A persistence daemon should never assume “no event” means “no change.”

4. **inotify rename semantics are inherently racy.** Rename pairs use `IN_MOVED_FROM`/`IN_MOVED_TO` with a cookie, but the pair is not guaranteed to be consecutive or atomically inserted; the `IN_MOVED_TO` may be missing if the target is outside the watched set, and matching is explicitly described as inherently racy. [inotify(7)](https://man7.org/linux/man-pages/man7/inotify.7.html)
   - Practical implication: treat renames as delete/create hints unless you can reconcile by inode/content snapshot. Do not build correctness around perfect cookie pairing.

5. **inotify path/name data can be stale or incomplete.** inotify identifies events by watch descriptor and optional child name; applications must maintain wd→path mappings, directory renames can invalidate many cached paths, and by processing time the file name may already be deleted or renamed. It also does not catch remote filesystem changes, pseudo-filesystems such as `/proc`, `/sys`, `/dev/pts`, or mmap/msync/munmap-caused accesses/modifications. [inotify(7)](https://man7.org/linux/man-pages/man7/inotify.7.html)
   - Practical implication: persist by verifying current filesystem state with `lstat`/directory scans before copying/deleting; exclude pseudo-filesystems and mount points explicitly.

6. **fanotify can monitor mounts/filesystems and supports permission events, but availability and privilege constraints are significant.** fanotify needs kernel `CONFIG_FANOTIFY`; permission handling also needs `CONFIG_FANOTIFY_ACCESS_PERMISSIONS`. It gained create/delete/move events only in Linux 5.1. Mount/filesystem monitoring can cover a tree race-free, but directory marks are not recursive and event queues can still overflow. [fanotify(7)](https://man7.org/linux/man-pages/man7/fanotify.7.html)
   - Practical implication: fanotify may be attractive for a privileged host-side daemon, but it is not a drop-in unprivileged container solution and still requires overflow recovery.

7. **fanotify privileges changed in Linux 5.13, but unprivileged mode is limited.** Before Linux 5.13 (and 5.10.220 backports), `fanotify_init()` required `CAP_SYS_ADMIN`. Since then, unprivileged callers can create only limited groups: no permission events, no unlimited queues/marks, must use file-handle reporting such as `FAN_REPORT_FID`, cannot mark mounts or filesystems, and receive limited pid/object info. `FAN_CLASS_CONTENT`, `FAN_CLASS_PRE_CONTENT`, `FAN_UNLIMITED_QUEUE`, `FAN_UNLIMITED_MARKS`, and `FAN_REPORT_TID` still require capabilities. [fanotify_init(2)](https://www.man7.org/linux/man-pages/man2/fanotify_init.2.html)
   - Practical implication: an in-container PaaS agent generally cannot rely on fanotify for whole-rootfs capture unless run with elevated caps/seccomp and the right kernel.

8. **Container seccomp profiles can block fanotify even when the kernel supports unprivileged fanotify.** Docker’s default seccomp model is a restrictive syscall policy, and a current Moby issue documents `fanotify_init` allowed only with `CAP_SYS_ADMIN` in the builtin profile, causing `EPERM` for limited unprivileged fanotify on Docker 28. [Docker seccomp docs](https://docs.docker.com/engine/security/seccomp/), [moby/moby#49756](https://github.com/moby/moby/issues/49756)
   - Practical implication: test on the exact runtime profile. If using fanotify, expect to need a custom seccomp profile and capabilities; also note Docker’s default seccomp docs list `open_by_handle_at` as blocked, which is relevant to resolving fanotify file handles. [Docker seccomp docs](https://docs.docker.com/engine/security/seccomp/)

9. **Node `fs.watch()` is a portability abstraction over inotify/libuv on Linux, not a reliable change log.** Node documents that `fs.watch()` uses inotify on Linux, is not 100% consistent, can be unavailable/unreliable on network filesystems or virtualized host filesystems, watches inodes on Linux/macOS, and may continue watching the old inode after a delete/recreate. `filename` may be `null`; event types are only `'rename'` and `'change'`. [Node.js fs docs](https://nodejs.org/api/fs.html#fswatchfilename-options-listener)
   - Practical implication: Node `fs.watch()` is acceptable as an invalidation signal, but not as the sole source of persistence truth.

10. **Node/libuv collapses Linux inotify detail and appears to drop kernel overflow events in the generic callback path.** libuv’s Linux backend maps `IN_ATTRIB|IN_MODIFY` to `UV_CHANGE` and all other masks to `UV_RENAME`; the watch lookup is by `e->wd`, so `IN_Q_OVERFLOW` (`wd == -1`) does not map to a watcher callback. [libuv Linux source](https://github.com/libuv/libuv/blob/v1.x/src/unix/linux.c), [libuv fs_event docs](http://docs.libuv.org/en/stable/fs_event.html)
   - Practical implication: if overflow detection is a hard requirement, Node `fs.watch()` may be the wrong abstraction. Consider a native inotify binding that exposes `IN_Q_OVERFLOW`, or schedule conservative periodic rescans regardless.

11. **Node recursive watching on Linux exists in modern Node, but it is implemented above kernel inotify, not by a kernel-recursive primitive.** Node added recursive `fs.watch({ recursive: true })` support for Linux in v19.1.0. Current Node source implements recursive watching by scanning entries and installing watchers, then reacting to events with stat/readdir logic. [Node v19.1.0 release](https://github.com/nodejs/node/commit/b322bf5314d9d1e34786bc4c42fa639e55ade080), [Node recursive_watch.js](https://github.com/nodejs/node/blob/main/lib/internal/fs/recursive_watch.js)
   - Practical implication: recursive `fs.watch` still inherits races around newly created/moved subtrees, watch limits, missed events, and abstraction loss. Validate behavior under your target Node version.

12. **`fsPromises.watch()` adds a userland queue overflow option, but that is not kernel inotify overflow handling.** Current Node docs expose `maxQueue` and `overflow: 'ignore'|'throw'` for the async iterator queue between iterations; this controls Node’s own event queueing, not the kernel’s `IN_Q_OVERFLOW`. [Node.js fsPromises.watch docs](https://nodejs.org/api/fs.html#fspromiseswatchfilename-options)
   - Practical implication: `overflow: 'throw'` is useful for backpressure in your async consumer, but it does not make the watcher complete.

13. **Overlay/container root filesystems add more correctness hazards.** Docker’s OverlayFS/overlay2 rootfs is a union of read-only lower layers and a writable upper layer; first writes can trigger full-file copy-up, deletes create whiteouts/opaque dirs, and directory `rename(2)` may return `EXDEV` unless source and destination are both in the top layer. Docker also recommends volumes for write-heavy workloads because they bypass storage-driver overhead. [Docker overlay2 docs](https://docs.docker.com/storage/storagedriver/overlayfs-driver/)
   - Practical implication: scan/copy logic must understand the container’s view, not just host layer paths, and should test copy-up/whiteout/EXDEV cases. Persisting the merged view from inside the container is simpler semantically than watching Docker internals under `/var/lib/docker`.

14. **A watcher-only design cannot be 100% correct.** The primary docs explicitly call out queue overflow, nonrecursive races, racy rename matching, stale names, coalescing, non-reported mmap/network/pseudo-fs events, and Node abstraction loss. fanotify mount/filesystem marks reduce the subtree-watch race but still can overflow and remain privilege/runtime constrained. [inotify(7)](https://man7.org/linux/man-pages/man7/inotify.7.html), [fanotify(7)](https://man7.org/linux/man-pages/man7/fanotify.7.html), [Node.js fs docs](https://nodejs.org/api/fs.html#fswatchfilename-options-listener)
   - Practical implication: for “correct persistence,” use watchers only to trigger reconciliation. Maintain a persistent manifest keyed by path plus stable metadata where possible (`dev`, `ino`, size, mtime/ctime, mode, xattrs if relevant), periodically crawl, crawl on startup and graceful shutdown, and full-rescan after overflow/errors/restarts. If strict exactly-once capture is required, move enforcement below the application layer (e.g., filesystem/snapshot layer, controlled write API, or privileged fanotify permission workflow) rather than relying on passive watchers alone.

## Gaps
- Exact behavior depends on kernel version, container runtime, seccomp profile, capabilities, storage driver/snapshotter, and whether the daemon runs inside the container, as a sidecar, or on the host. Validate with an integration test matrix that deliberately triggers burst overflow, subtree creation-with-immediate-writes, cross-directory renames, delete/recreate inode replacement, overlay copy-up, whiteouts, bind mounts, and daemon restart during writes.
- Node’s public API does not document kernel `IN_Q_OVERFLOW` propagation; libuv source suggests it is not surfaced through `fs.watch()`. If explicit overflow handling is required, prototype with a native inotify/fanotify binding or a small helper process and verify observed errors under load.
